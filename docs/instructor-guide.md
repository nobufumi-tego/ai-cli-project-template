# 講師担当ガイド
## PC 初心者向け初級講座でこのリポジトリを使う場合の責任分担

対象：スマートフォンしか使ったことがない大学院生への初級ターミナル講座（複数 AI CLI の活用を含む）
使用リポジトリ：ai-cli-project-template

---

## このドキュメントの読み方

このリポジトリは「開発環境のテンプレート」であり、「授業教材」ではありません。
以下の表を起点に、**リポジトリが担う部分**と**講師が担う部分**を明確に分けて設計してください。

| 担当 | 内容 |
|---|---|
| **リポジトリが提供するもの** | コマンド手順・環境診断・エラー時回復手順・AI サポート導線 |
| **講師が担うべきもの** | 概念説明・動機付け・授業設計・チェックポイント・トラブル対応体制 |

---

## 第 1 部：授業開始前に講師が準備すること

### 1-1. 使用環境を 1 つに絞る

このリポジトリは Windows 11 / macOS / Linux に対応しますが、
**受講者全員が同一 OS を使う状態にすること**が最重要です。

OS が混在すると、ひとつのコマンドで複数の説明が必要になり初心者が混乱します。

| 推奨構成 | 条件 |
|---|---|
| Windows 11（PowerShell のみ）| 大学支給 PC が Windows の場合。WSL2 は使わない |
| macOS（Terminal のみ）| 学生が Mac 持参の場合 |
| WSL2（Ubuntu）| 情報科学系でのちに Linux を使う前提がある場合のみ |

> **WSL2 は初回講座には推奨しません。**
> WSL2 を有効化する手順が加わり、「Windows の中に Linux がある」という概念説明が必要になります。
> それ自体が 1 コマ分の講義になります。

---

### 1-2. 自分の手で全手順を通して実行する

受講生と同一 OS・同一状態（ツール未インストール）で `beginner-guide.md` を最初から最後まで実行し、
**実際にかかる時間**と**詰まりやすい箇所**を把握してください。

確認すべき具体的な点：

- `python`（Windows）または `python3`（macOS/Linux）コマンドが通るか
- `uv_setup/install.bat` ダブルクリック時に「Windows によって PC が保護されました」が出るか
- `uv` インストール後にターミナルを開き直さないと `uv --version` が通らないか
- `uv run python scripts/init-project.py` の対話で日本語入力が文字化けしないか
- `check-setup.py` の自動言語判定が環境のロケール設定と一致しているか

---

### 1-3. ML フル環境（PyTorch 等）が必要かを判断する

`uv_setup/install.sh` および `install.ps1` は uv のほかに
**PyTorch・JupyterLab 等（約 2GB）を `uv_setup/.venv/` にインストール**します。

初級講座の第 1 回目で ML ライブラリが必要になることはほぼありません。
**最初の授業では uv 本体のみのインストールを推奨します。**

uv のみをインストールするコマンド：

```bash
# macOS / Linux（WSL2 含む）
curl -LsSf https://astral.sh/uv/install.sh | sh

# Windows（PowerShell）
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

ML ライブラリが必要になった時点で `uv_setup/install.sh` を使うよう案内してください。

---

### 1-4. GitHub の設定を行う

受講生がリポジトリを利用できるようにするために以下を設定してください。

1. GitHub リポジトリの Settings → 「Template repository」にチェック
2. About（説明文）を設定する
3. 受講生が GitHub アカウントを持っていない場合は事前に作成させる
4. GitHub Classroom を使う場合は Assignment を作成しておく

## 第 2 部：講師が授業中に担当すること

### 2-1.【最重要】概念の説明（ターミナルを開く前に）

このリポジトリの `beginner-guide.md` はいきなり「ターミナルを開いて」から始まります。
スマートフォンしか使ったことがない学生には、**その前に以下の概念を図で説明**してください。
これをスキップすると、以降の手順がすべて「謎の呪文を打ち込む作業」になります。

---

#### 概念 A：ファイルに「場所」がある

スマートフォンのアプリはファイルの場所をユーザーから隠します。
PC では **「どのフォルダの中に入っているか」がすべての操作の前提**になります。

説明のステップ：

1. まずファイルエクスプローラー（Windows）または Finder（macOS）で
   「デスクトップ」→「ドキュメント」→「Downloads」の階層を実際に見せる
2. 同じ構造がターミナルでは文字で表現されることを示す
   - `C:\Users\あなたの名前\Documents` ← Windows
   - `/home/あなたの名前/documents` ← Linux / WSL2
3. 「今自分がどこにいるか」を確認するコマンドを最初に教える
   - `cd`（Windows）/ `pwd`（macOS / Linux）

**伝えたいメッセージ：**
「ターミナルは、マウスの代わりにキーボードでフォルダを移動するツールです。」

---

#### 概念 B：ターミナルは「テキストで動かすアプリランチャー」

「黒い画面に文字を打つ」というイメージが学生の心理的障壁になります。

説明例：
「スマートフォンの音声アシスタント（Siri・Google アシスタント）は声でアプリを動かします。
ターミナルはそれをテキストでやるものです。
コマンドは短縮された英語の指示文です。`cd` は "Change Directory"（フォルダを移動する）の略です。」

---

#### 概念 C：なぜ Git を使うのか

「コードを GitHub に置く」という行為の意味を説明してください。

説明例：
「論文の下書きを Google ドライブに保存して、変更履歴を残しながら共同編集するのと同じことを、
プログラムのコードでやるツールが Git です。
3 年後に今日の実験を再現したいとき、そのときのコードを取り出せます。」

---

#### 概念 D：なぜ uv を使うのか

説明例：
「Python ライブラリのバージョンは毎年変わります。
今日書いたコードが、2 年後の環境で動かなくなることがあります。
uv はそのときの環境をそのまま記録・再現するツールです。
研究の再現性に直結します。」

---

### 2-2. 各ステップの「なぜ」を補足する

`beginner-guide.md` の各ステップには「何をするか」は書いてありますが、
「なぜこの手順が必要か」は最小限です。
講師は各ステップの冒頭で以下の補足を口頭で行ってください。

| ステップ | 講師が口頭で補足すること |
|---|---|
| Step 1：Git インストール | 「これがないと次のダウンロードができない。インストール確認の `--version` パターンはすべてのツールで使える」 |
| Step 2：クローン | 「`git clone` でダウンロードして `cd` で中に入る、この 2 つは常にセットで使う」 |
| Step 3：uv インストール | 「今日は uv 本体だけ。ML ライブラリは別の回にまとめて入れる」 |
| Step 4：プロジェクト初期化 | 「ここで生成される AGENTS.md が AI に対するプロジェクトの説明書になる」 |
| Step 5：VS Code | 「ターミナルでもコードは書けるが、VS Code を使うと色がついて読みやすくなる」 |
| Step 6：AI CLI | 「今日は任意。インストールできなくても次回以降の演習には支障ない」 |

---

### 2-3. チェックポイントを設ける

`beginner-guide.md` には「期待される出力」が記載されていますが、
学生が自力で「合っているか」を判断するのは難しい場合があります。

**各ステップの終わりに「全員に画面を見せてもらう」時間を設けてください。**

推奨チェックポイント：

| 場所 | 確認内容 |
|---|---|
| Step 1 終了後 | `git --version` の出力が画面に出ているか |
| Step 3 終了後 | `uv --version` の出力が画面に出ているか |
| Step 4 終了後 | `AGENTS.md` と `README.md` が生成されているか（VS Code または `ls` で確認） |
| 全体終了後 | `check-setup.py` を実行して「必須ツールはすべて揃っています」と出るか |

---

### 2-4. トラブル対応のエスカレーション設計

以下の 3 段階を事前に受講生に伝えておいてください。

```
段階 1（5 分以内）: 自分で解決を試みる
  → エラーメッセージをそのまま Claude / Gemini / ChatGPT に貼り付ける
  → beginner-guide.md の「よくあるエラーと解決策」を確認する

段階 2（5〜10 分）: TA または隣の学生に確認する
  → 「期待される出力」と自分の出力を見比べてもらう

段階 3（10 分以上解決しない）: 講師に報告する
  → check-setup.py の出力をコピーして見せる
```

**講師が把握しておくべきよくある詰まりポイント（Windows 11）：**

| 症状 | 原因 | 対処 |
|---|---|---|
| `python` を打つと Microsoft Store が開く | Python 未インストール | `winget install Python.Python.3.12` |
| `uv --version` が「コマンドが見つかりません」 | ターミナルを開き直していない | ターミナルを閉じて再起動 |
| `install.bat` ダブルクリックで警告が出る | SmartScreen の警告 | 「詳細情報」→「実行」を案内 |
| 日本語が文字化けする | コードページが UTF-8 でない | `chcp 65001` を実行してから再試行 |
| `uv run` が動かない | PATH が通っていない | `$env:PATH` に `~\.local\bin` を追加 |

## 第 3 部：複数 AI CLI チュートリアルとしての設計

このリポジトリは Claude Code・Gemini CLI・Codex CLI の 3 ツールに対応しており、
「AGENTS.md を 1 つ書くだけで全ツールに反映される」という体験を提供できます。
ただし、**PC 初心者が最初から 3 つを同時に扱うことは現実的ではありません。**
以下の障壁と設計方針を把握した上で段階的に導入してください。

---

### 3-1. 複数 AI CLI 導入の障壁

#### 障壁 1（最大）：認証と費用が 3 種類ある

| CLI | 認証方法 | 費用 |
|---|---|---|
| Claude Code | Anthropic アカウント + ブラウザ OAuth | 有料（Pro プランまたは API キー） |
| Gemini CLI | Google アカウント + ブラウザ OAuth | 無料枠あり |
| Codex CLI | OpenAI API キー | 従量課金（クレジットカード必須） |

学生に「3 種類のアカウントを作り、うち 2 つは課金設定を行う」ことを求めるのは、
ターミナル操作よりも高い障壁になります。
**Codex CLI はクレジットカードが必須のため、学生には強制しないでください。**

#### 障壁 2：インストール層が多すぎる

AI CLI を動かすまでに通過する層：

```
Git → uv → Python → Node.js（npm を同梱）→ Claude Code（または Gemini CLI）
```

Node.js は AI CLI のためだけに必要なツールです（npm は Node.js に同梱されており、別途インストールは不要）。
「なぜ Node.js が必要なのか」を説明なしに進めると、初心者には「謎の手順」になります。
口頭で「AI CLI は Node.js で動くアプリだから、その実行環境として必要」と補足してください。

#### 障壁 3：AGENTS.md の価値は 2 つ目の CLI を使うまで見えない

「1 ファイルで 3 ツールに対応」というメリットは、
**1 つ目の CLI を使っている間はまったく体感できません。**
2 つ目の CLI を起動して「あ、同じ指示が通じる」と気づく瞬間に初めて意味を持ちます。
この体験を意図的に設計することが、講師の最重要タスクです。

---

### 3-2. 推奨する導入順序

```
第 1 CLI：Claude Code または Gemini CLI（どちらか一方）
  理由：ブラウザ認証のみで動く。課金設定が不要または無料枠がある。
  目標：「AGENTS.md を書いたら AI がプロジェクトを理解した」体験

第 2 CLI：上記と異なるもう一方（Claude Code または Gemini CLI）
  理由：認証フローが似ており、追加の概念説明が少ない。
  目標：「AGENTS.md を変えていないのに別の AI も理解している」体験
  ← ここで初めて「1 ファイル管理」の意義が腑に落ちる

第 3 CLI（任意）：Codex CLI
  条件：OpenAI API キーを自分で用意できる受講生のみ
  時期：第 2 CLI が安定して使えるようになってから
```

**第 2 CLI を追加する回は、AGENTS.md の「1 ファイル管理」を体験させる専用回として設計してください。**
具体的な演習例：
1. 第 1 CLI で「このプロジェクトのアーキテクチャを説明して」と聞く
2. AGENTS.md を変えずに第 2 CLI を起動して同じ質問をする
3. 答えが一致することを確認する

---

### 3-3. 「なぜ複数の CLI を使うのか」の説明

口頭で補足すべき動機付け：

「Claude Code・Gemini CLI はどちらも同じような作業ができますが、
得意なことや使い勝手が少しずつ違います。
どれが自分の研究スタイルに合うかは実際に使ってみないとわかりません。
このリポジトリの AGENTS.md は、どのツールを使っても同じ指示が通るように設計されています。
ツールを乗り換えても、プロジェクトの設定を書き直す必要がありません。」

## まとめ：講師の主担当領域

```
リポジトリが自動化できないもの：

1. 「なぜこのツールを使うのか」の動機付け説明
2. ファイルシステム・パス・ターミナルの概念図解
3. 受講者全員の環境を同一にそろえる意思決定（OS 選択）
4. 各ステップの「期待される出力」の全員確認
5. 詰まった学生のエスカレーション対応
6. AI CLI の費用・認証方法の事前説明と導入順序の決定
7. 「2 つ目の CLI で AGENTS.md がそのまま使える」体験の設計
8. Codex CLI を強制しない判断（クレジットカード不要の 2 ツールで完結させる選択）
9. 概念説明スライド等の視覚教材の作成

リポジトリが提供するもの（再掲）：

- コマンドの手順書（beginner-guide.md）
- 環境診断（check-setup.py）
- AGENTS.md の 1 ファイル管理構造（3 ツール対応済み）
- エラー時の回復手順（recovery.md）
- AI チャットボット一次対応の導線（SUPPORT.md・beginner-guide 末尾）
```
